# Basic-bot

A minimal, plugin-based Telegram bot skeleton built with [Pyrogram](https://docs.pyrogram.org/) (kurigram fork), Motor/MongoDB-ready, and a built-in health check webserver for deployment on platforms like Koyeb.

Import this as a starting point and start adding commands/features as standalone plugin files — no need to touch `bot.py`.

## Structure

```
Basic-bot-main/
├── bot.py              # Client setup + entrypoint — rarely needs editing
├── config.py           # Env-based configuration
├── webserver.py         # aiohttp health check server (for Koyeb/uptime pings)
├── plugins/             # Your handlers live here — auto-loaded on startup
│   ├── __init__.py
│   └── start.py         # Example: /start command
├── requirements.txt
├── Dockerfile
└── README.md
```

## How it works

`bot.py` creates a single `Client` instance and points it at the `plugins/` folder:

```python
plugins=dict(root="plugins")
```

On startup, Pyrogram automatically imports every `.py` file inside `plugins/`. This means any handler defined with `@Client.on_message(...)` (or `@Client.on_callback_query`, `@Client.on_edited_message`, etc.) in that folder is registered automatically — just by existing there.

**You never need to import your plugin files manually, and you never need to edit `bot.py` to add a new command.**

## Adding a new command

1. Create a new file in `plugins/`, e.g. `plugins/help.py`.
2. Use the class-level decorator pattern:

```python
from pyrogram import Client, filters
from pyrogram.types import Message


@Client.on_message(filters.command("help") & filters.private)
async def help_handler(client: Client, message: Message):
    await message.reply("This is the help command.")
```

3. Restart the bot. That's it — no other file needs to change.

### Guidelines for plugin files

- One feature area per file (e.g. `start.py`, `broadcast.py`, `settings.py`) — keeps things easy to navigate as the bot grows.
- Always import `Client` and `filters` from `pyrogram`, and `Message` (or `CallbackQuery`, etc.) from `pyrogram.types`.
- Always use `@Client.on_message(...)` / `@Client.on_callback_query(...)` — **not** `@app.on_message(...)`. The class-level decorator is what makes the plugin auto-loading work.
- Filenames don't matter to Pyrogram, but keep them descriptive.
- Avoid naming collisions between handler function names across files (Python doesn't care, but it helps readability).

## Configuration

All config is read from environment variables in `config.py`:

| Variable    | Description                          |
|-------------|---------------------------------------|
| `API_ID`    | Telegram API ID from [my.telegram.org](https://my.telegram.org) |
| `API_HASH`  | Telegram API hash from my.telegram.org |
| `BOT_TOKEN` | Bot token from [@BotFather](https://t.me/BotFather) |

Add any new config values you need the same way — read them in `config.py` and import from there in your plugins, rather than reading `os.environ` directly in plugin files.

## Database (MongoDB / Motor)

`requirements.txt` already includes `motor` and `pymongo`. To wire up a database:

1. Add a `MONGO_URI` variable to `config.py`.
2. Create a small `db.py` (or `database/` package) that initializes the Motor client once and exposes it, e.g.:

```python
import motor.motor_asyncio
from config import MONGO_URI

mongo_client = motor.motor_asyncio.AsyncIOMotorClient(MONGO_URI)
db = mongo_client["your_db_name"]
```

3. Import `db` from your plugin files as needed. Keep the client creation lazy or module-level (initialized once at import time), not per-request.

## Webserver / Health check

`webserver.py` runs a lightweight `aiohttp` server on port `8080` alongside the bot, exposing:

- `GET /` — basic status JSON
- `GET /health` — plain `OK` (useful for uptime monitors / Koyeb health checks)

It starts automatically in `BroadcastBot.start()` and shuts down cleanly in `stop()`. You generally don't need to touch this file unless you want to add more routes.

## Running locally

```bash
pip install -r requirements.txt
export API_ID=xxxx
export API_HASH=xxxx
export BOT_TOKEN=xxxx
python bot.py
```

## Deploying on Koyeb (or similar)

The included `Dockerfile` builds and runs the bot as-is:

```bash
docker build -t basic-bot .
docker run -e API_ID=xxxx -e API_HASH=xxxx -e BOT_TOKEN=xxxx -p 8080:8080 basic-bot
```

Set `API_ID`, `API_HASH`, and `BOT_TOKEN` as environment variables/secrets in your Koyeb service settings. The exposed port `8080` is used for the health check endpoint.

## Notes / Gotchas

- Don't rename the `plugins` folder without also updating the `plugins=dict(root="plugins")` line in `bot.py`.
- `__pycache__/` folders are auto-generated by Python and safe to ignore — see `.gitignore`.
- If a plugin file has a syntax error, Pyrogram may fail silently or throw at startup depending on version — check logs on boot if a new command doesn't show up.
